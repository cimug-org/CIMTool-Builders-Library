# csharp-ef-rdfs.xsl

This builder currently ships with CIMTool.

## Builder Description

> **Important:** This builder is specifically intended for use with profiles designed to generate RDFS as their primary schema artifact. For such profiles, defined using the formal CIM profiling approach for RDFS schemas, this builder generates a complementary C# Entity Framework Core persistence layer that is structurally aligned with the RDFS schema. It is designed to be used alongside the companion **[sql-rdfs-ansi92.xsl](../sql-rdfs-ansi92/builder.md)** and **[jpa-rdfs.xsl](../jpa-rdfs/builder.md)** builders.

The **[csharp-ef-rdfs.xsl](csharp-ef-rdfs.xsl)** builder generates a C# source file containing [Entity Framework Core](https://learn.microsoft.com/en-us/ef/core/) entity classes corresponding to the classes, attributes, and associations defined in a **CIMTool** profile. The generated classes include `[Table]`, `[Key]`, `[MaxLength]`, and `[ForeignKey]` annotations compatible with EF Core's data annotations approach, as well as a `DbContext` class and a `DbContextBase` base class for use in .NET applications.

A key distinction from earlier C# builders is the RDFS-schema-aware primary key strategy. For all profile classes in the `IdentifiedObject` hierarchy, `mRID` is used as the natural `[Key]` — a `[MaxLength(100)]` string property that mirrors the primary key generated by the companion **[sql-rdfs-ansi92.xsl](../sql-rdfs-ansi92/builder.md)** builder. For non-`IdentifiedObject` classes (such as `CurveData`, `RegularTimePoint`, and `TapChangerTablePoint`), a surrogate UUID `id` string property is emitted as the `[Key]` together with a heuristic `[Index(..., IsUnique = true)]` annotation spanning all required columns. String properties receive `[MaxLength(255)]` annotations, `anyURI` properties receive `[MaxLength(2048)]`, consistent with the column lengths emitted by the companion SQL builder. C# inheritance is fully supported.

This builder is designed to be used in conjunction with the companion **[sql-rdfs-ansi92.xsl](../sql-rdfs-ansi92/builder.md)** and **[jpa-rdfs.xsl](../jpa-rdfs/builder.md)** builders, which apply the identical primary key detection logic to generate a structurally aligned SQL DDL schema and JPA Java persistence layer respectively.

## Builder NTE Configuration

Given this builder ships with CIMTool the below table highlights the existing NTE (Name/Type/Extension) settings for the builder.  We strongly recommend that you do not import customized versions of this shipped builder over the original in your local CIMTool installation. Doing so will force regeneration of any existing builder-generated artifacts for projects in your workspace that have this builder enabled.

The recommended best practice is to rename the builder file  (from ```csharp-ef-rdfs.xsl``` to ```custom-csharp-ef-rdfs.xsl``` for example) before importing. On import this will be interpreted as a new builder and you will be allowed to enter values in the Type and Extension fields.

>*NOTE: </br>CIMTool requires that file extensions be unique and will prevent you from entering an extension already assigned to a builder. This is because an artifact's name is derived by concatenating the base name of the CIMTool ```.owl``` profile with the file extension assigned to the builder. Therefore, a unique file extension must be assigned to each builder when imported. The file extension for a builder can be modified later from within the "Maintain XSLT Transform Builders" screen.*

A blank Includes File Name row indicates that no XSLT include file is required for this builder.

| Field | Value |
|---|---|
| Builder XSLT Name: | `csharp-ef-rdfs.xsl` |
| Includes File Name: |  |
| Transform Builder Type: | `TEXT` |
| File Extension of Generated Files: | `csharp-ef-rdfs.cs` |
